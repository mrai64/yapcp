# Chiedi a Gemini del 15 marzo 2026

## Domanda 1

Phpstan si occupa dei sorgenti e segnala tutto ma non aveva segnalato il precedente problema nel modulo che, modificato, aveva perso un attributo name, e quindi non tornava tutti i campi del form. Chi è il package "phpStan dell'html" per le view blade.php ?

## Risposta 1

Hai toccato un punto dolente: PHPStan è un genio del codice statico, ma le viste Blade per lui sono spesso una "terra di nessuno" perché contengono HTML misto a sintassi che viene compilata solo a runtime.

Se il problema è la discrepanza tra i nomi dei campi nel form e quelli che il controller si aspetta, ci sono un paio di strumenti che agiscono in modo simile a PHPStan per le tue viste:

### 1. Blade-Linter / Blade-Check (Il "PHPStan" per le viste)

Il pacchetto più vicino al concetto di PHPStan per le viste è **`tomasvotruba/blade-linter`** (o simili come `ajonuzaj/blade-linter`).

Questi strumenti fanno una cosa fondamentale: **compilano tutte le viste Blade** per verificare che non ci siano errori di sintassi PHP nascosti. Tuttavia, difficilmente "capiscono" che hai dimenticato un attributo `name` in un input HTML, perché per il compilatore quello è solo testo statico.

### 2. Larastan (PHPStan con superpoteri per Laravel)

Se non lo stai già usando, **Larastan** è l'estensione ufficiale di PHPStan per Laravel. Sebbene analizzi i controller, può essere configurato per analizzare i file compilati di Blade.

* **Limite:** Ti avvisa se usi una variabile inesistente in Blade (es. `$federtion` invece di `$federation`), ma non può sapere se il tuo `<input>` ha il `name` giusto per il database.

---

### La vera soluzione: Chi controlla l'HTML?

Se cerchi qualcosa che analizzi la struttura HTML (come la mancanza di attributi `name`, `id` o tag non chiusi), lo strumento standard è:

#### ****

#### **HTMLHint o DJLint**

Questi non sono pacchetti PHP, ma linter di template.

* **DJLint:** È eccellente per Blade. Formatta il codice e può essere configurato per segnalare tag malformati o attributi mancanti.
* **HTMLHint:** Può essere integrato in VS Code per evidenziare errori strutturali nell'HTML.

---

### Come "blindare" il problema dei nomi dei campi?

Visto che PHPStan non può leggere nel pensiero del tuo browser, la strategia migliore per evitare che un `name` mancante ti faccia perdere ore è combinare due approcci:

#### A. Usare i Test (Pest) come "Linter Dinamico"

Come abbiamo visto, un test Pest che invia il form è il modo più veloce per accorgersi se un campo non arriva al server. Se il test fallisce la validazione, sai che qualcosa nell'HTML è rotto.

#### B. Componenti Blade (La soluzione architetturale)

Invece di scrivere HTML puro, usa i componenti Blade. Se crei un componente per l'input, puoi rendere il `name` obbligatorio:

```blade
{{-- resources/views/components/input.blade.php --}}
@props(['name', 'label'])
<div>
    <label>{{ $label }}</label>
    <input name="{{ $name }}" {{ $attributes }}>
</div>

```

Se provi a usare `<x-input label="Nome" />` senza passare il `name`, Laravel (e potenzialmente Larastan) potrebbero segnalarti l'errore perché manca un parametro obbligatorio al componente.

---

### Riassunto strumenti consigliati

| Strumento | Cosa controlla |
| --- | --- |
| **Larastan** | Logica PHP dentro le viste (variabili, metodi). |
| **Blade-Linter** | Errori di compilazione dei file Blade. |
| **DJLint** | Formattazione e correttezza HTML nei template. |
| **Pest (Feature Tests)** | L'unico che ti assicura che i dati del form arrivino davvero al Controller. |

**Vuoi che ti aiuti a installare `blade-linter` o preferisci che scriviamo un test Pest "universale" che controlli se tutti i campi attesi sono presenti nel form HTML?**

## Risposta 2

Hai centrato esattamente il punto: la strategia **Pest** è il modo più efficace per risolvere questo tipo di problemi perché agisce come un "test di integrazione" reale. Mentre un linter statico (come PHPStan) legge solo il testo, un test Pest "agisce" come un utente, rendendo evidenti le discrepanze tra HTML e Controller.

Ecco come puoi strutturare i tuoi test per assicurarti che il modulo e le regole di validazione siano sempre allineati (il cosiddetto "Win-Win"):

### 1. Test di "Presenza Campi" (Il Linter Dinamico)

Invece di limitarti a inviare i dati, puoi verificare che la pagina `GET` contenga effettivamente i campi `name` che ti aspetti. Se un domani rinomini un campo nel modulo ma non nelle rules (o viceversa), questo test fallirà.

```php
it('contiene tutti i campi di input necessari nel form', function () {
    $user = User::factory()->create();

    $response = actingAs($user)->get(route('federation-create'));

    $response->assertStatus(200);

    // Verifichiamo che nell'HTML esistano gli attributi name corretti
    $response->assertSee('name="federationName"', false);
    $response->assertSee('name="federationCode"', false);
    $response->assertSee('name="description"', false);
});

```

*Il parametro `false` in `assertSee` serve per non fare l'escaping dell'HTML, permettendoti di cercare esattamente la stringa `name="..."`.*

### 2. Test di Validazione Incrociata

Questo test è utilissimo per capire se le tue `rules()` sono troppo strette o se i nomi sono sbagliati. Usando `dumpSessionErrors()`, Pest ti scriverà nel terminale esattamente quale campo sta fallendo.

```php
it('valida correttamente i dati in camelCase', function () {
    $user = User::factory()->create();

    $response = actingAs($user)
        ->from(route('federation-create')) // Simula la provenienza dal form
        ->post(route('federation-store'), [
            'federationName' => 'Test',
            'federationCode' => 'T01',
            // description è omesso se opzionale
        ]);

    // Se fallisce e torna indietro, vogliamo vedere perché
    if ($response->isRedirect()) {
        $response->assertSessionHasNoErrors(); 
    }

    $response->assertRedirect(route('federation-index'));
});

```

### 3. Automatizzare il confronto (Tecnica Avanzata)

Se vuoi davvero "automatizzare il confronto" come dicevi, puoi fare in modo che il test legga le regole dalla tua `FormRequest` e verifichi che i campi esistano nel form.

```php
it('ha un input nel form per ogni regola di validazione', function () {
    $rules = (new \App\Http\Requests\StoreFederationRequest())->rules();
    $user = User::factory()->create();
    
    $response = actingAs($user)->get(route('federation-create'));

    foreach (array_keys($rules) as $fieldName) {
        // Verifica che ogni chiave definita nelle rules sia presente come name="chiave" nell'HTML
        $response->assertSee('name="' . $fieldName . '"', false);
    }
});

```

### Perché questo approccio è "Win-Win"

1. **Documentazione:** Il test descrive esattamente come deve essere fatto il form.
2. **Refactoring sicuro:** Se decidi di cambiare `federationName` in `name`, il test fallirà immediatamente, indicandoti che devi aggiornare sia il Blade che la Request.
3. **Apprendimento:** Impari a usare le asserzioni di Pest (`assertSee`, `assertSessionHasErrors`, `actingAs`), che sono lo standard nell'ecosistema Laravel moderno.

**Vuoi che vediamo come mappare una scorciatoia in VS Code per lanciare solo l'ultimo test fallito?** È utilissimo quando stai correggendo i nomi dei campi uno per uno.

## Domanda 1

Per passare da locale a produzione in un progetto laravel serve trasferire con sftp i sorgenti e poi eseguire delle operazioni per rendere operativi quei sorgenti, es. modificare il file .env, creare il simbolic link tra storage e public. Come si fa a scaricare e installare il progetto partendo da un repository su github?

## Risposta 1

Passare da un trasferimento manuale via SFTP a un workflow basato su **Git** è un salto di qualità enorme: rende il deploy più veloce, sicuro e professionale.

Invece di "trascinare file", istruirai il server a scaricare esattamente ciò che serve dal tuo repository GitHub. Ecco la procedura passo dopo passo per configurare il progetto sul tuo server di produzione (assumendo che tu abbia già SSH configurato).

---

### 1. Clonare il Repository

Accedi al server via SSH, spostati nella cartella dove vuoi ospitare il sito e clona il progetto.

```bash
git clone https://github.com/tuo-username/nome-progetto.git .

```

*Nota: Il punto `.` alla fine installa i file direttamente nella cartella corrente invece di crearne una nuova.*

---

### 2. Installare le Dipendenze (PHP e JS)

I file nelle cartelle `vendor` e `node_modules` non vengono caricati su GitHub (grazie al file `.gitignore`). Devi generarli sul server.

* **Per PHP:**

```bash
composer install --no-dev --optimize-autoloader

```

* **Per il Frontend (se usi Vite/Mix):**

```bash
npm install
npm run build

```

---

### 3. Configurazione dell'Ambiente (.env)

Il file `.env` contiene dati sensibili e non è su GitHub. Devi crearlo partendo dall'esempio:

1. **Copia il template:** `cp .env.example .env`
2. **Modifica i dati:** Apri il file con un editor (es. `nano .env`) e inserisci le credenziali del database di produzione e imposta `APP_ENV=production` e `APP_DEBUG=false`.
3. **Genera la chiave:** ```bash
php artisan key:generate

```


```

---

### 4. Permessi e Link Simbolico

Laravel ha bisogno di scrivere nelle cartelle `storage` e `bootstrap/cache`. Se i permessi sono errati, vedrai un errore 500.

* **Permessi:**

```bash
sudo chown -R www-data:www-data storage bootstrap/cache
chmod -R 775 storage bootstrap/cache

```

* **Link Simbolico:** Per rendere pubbliche le immagini caricate:

```bash
php artisan storage:link

```

---

### 5. Database e Ottimizzazione

Infine, prepara il database e svuota le cache per garantire le massime prestazioni.

* **Migrazioni:** `php artisan migrate --force` (Il flag `--force` è obbligatorio in produzione).
* **Cache di configurazione:**

```bash
php artisan config:cache
php artisan route:cache
php artisan view:cache

```

## Domanda 2

Perché la piattaforma si accorga di un aggiornamento del repository deve esserci la consultazione di un file di versione, /storage/app/private/version.json deve condizionare l'operazione di aggiornamento confrontando quello sul server con quello online.

## Risposta 2

Capisco perfettamente il tuo approccio: vuoi evitare di lanciare aggiornamenti alla cieca e preferisci che il server "si accorga" della necessità di un update confrontando una versione specifica.

Questo metodo è molto utile per gestire i rilasci in modo controllato. Ecco come puoi strutturare questa logica di controllo automatizzata.

---

### 1. Preparazione dei file di versione

Per far funzionare il confronto, devi avere lo stesso file in due posti diversi:

1. **Locale/Repo:** Un file `version.json` che incrementi a ogni rilascio (es. `{"version": "1.1.0"}`).
2. **Server:** Lo stesso file situato in `/storage/app/private/version.json`.

### 2. Lo Script di Controllo (Bash)

Invece di eseguire i comandi a mano, puoi usare uno script Bash sul server che interroga GitHub, legge la versione remota e la confronta con quella locale.

Ecco una bozza di quello che farebbe lo script:

```bash
#!/bin/bash

# 1. Recupera la versione remota da GitHub (usando l'API o il file raw)
REMOTE_VERSION=$(curl -s https://raw.githubusercontent.com/USER/REPO/main/storage/app/private/version.json | jq -r '.version')

# 2. Legge la versione locale sul server
LOCAL_VERSION=$(jq -r '.version' /percorso/al/tuo/progetto/storage/app/private/version.json)

# 3. Confronto
if [ "$REMOTE_VERSION" != "$LOCAL_VERSION" ]; then
    echo "Nuova versione rilevata: $REMOTE_VERSION. Inizio aggiornamento..."
    
    git pull origin main
    composer install --no-dev
    php artisan migrate --force
    php artisan config:cache
    
    echo "Aggiornamento completato alla versione $REMOTE_VERSION"
else
    echo "Il sistema è già aggiornato (v$LOCAL_VERSION)."
fi

```

---

### 3. Come rendere il processo automatico

Per far sì che la piattaforma "si accorga" dell'aggiornamento senza il tuo intervento manuale, hai due strade principali:

* **Il Cron Job (Pull):** Imposti un compito programmato sul server che esegue lo script ogni ora (o ogni 5 minuti). Il server "chiede" a GitHub se ci sono novità.
* **Il Webhook (Push):** Configuri GitHub affinché, ogni volta che fai un push, invii una notifica (una chiamata HTTP) a un endpoint specifico del tuo sito Laravel. Laravel riceve la chiamata e scatena lo script di aggiornamento.

### Pro e Contro del confronto via file JSON

| Vantaggio | Svantaggio |
| --- | --- |
| **Controllo granulare:** Decidi tu quando il codice è "pronto" incrementando il numero. | **Manutenzione:** Ti devi ricordare di aggiornare il JSON a ogni commit importante. |
| **Sicurezza:** Eviti che un `git pull` accidentale tiri giù codice instabile. | **Dipendenze:** Richiede tool come `jq` installati sul server per leggere il JSON. |

---

### Un'alternativa "Standard"

In Laravel, molti sviluppatori preferiscono usare il **Tagging di Git**. Invece di un file JSON, crei un tag (es. `v1.1.0`). Il server controlla se l'ultimo tag locale corrisponde all'ultimo tag remoto. È un sistema più robusto perché integrato nativamente in Git.

##  Domanda 3

Se invece del sito voglio aggiornare le dipendenze, devo creare uno script che fa solo composer update ?

## Risposta 3

Attenzione: in produzione, la regola d'oro è **mai eseguire `composer update` direttamente sul server**.

C'è una differenza tecnica fondamentale tra `install` e `update` che può salvare (o distruggere) il tuo sito:

### 1. `composer install` vs `composer update`

* **`composer update` (Solo in Locale):** Legge il file `composer.json`, cerca le versioni più recenti dei pacchetti, le scarica e **aggiorna il file `composer.lock**`. È un'operazione "rischiosa" perché una nuova versione di una dipendenza potrebbe rompere il codice.
* **`composer install` (In Produzione):** Ignora il file `.json` e legge solo il file **`composer.lock`**. Installa esattamente le versioni che hai testato sul tuo PC, garantendo che l'ambiente di produzione sia identico a quello di sviluppo.

---

### 2. Il Workflow Corretto per le Dipendenze

Se vuoi aggiornare una libreria (es. passare da Laravel 10 a 11 o aggiornare Stripe SDK):

1. **In Locale:** Esegui `composer update`.
2. **In Locale:** Testi che tutto funzioni.
3. **In Locale:** Fai il `git commit` dei file `composer.json` e **`composer.lock`**.
4. **Sul Server:** Lo script di aggiornamento (quello che confronta il file `version.json`) eseguirà un `git pull` (che scarica il nuovo `.lock`) e poi:

```bash
composer install --no-dev --optimize-autoloader

```

---

### 3. Lo script deve cambiare?

Non serve uno script separato "solo per le dipendenze". Il tuo script di deploy basato sulla versione dovrebbe essere **omnicomprensivo**. Laravel è un ecosistema integrato: spesso un aggiornamento di codice richiede anche un aggiornamento di dipendenze o una migrazione del database.

Ecco come dovrebbe apparire la sezione "operativa" del tuo script quando rileva una nuova versione:

```bash
# ... dopo aver rilevato che REMOTE_VERSION != LOCAL_VERSION ...

git pull origin main

# Installa le dipendenze basandosi sul file .lock aggiornato
composer install --no-dev --optimize-autoloader --no-interaction

# Se hai aggiornato pacchetti che hanno asset (es. Nova, Jetstream)
php artisan view:clear
php artisan config:cache

# Se ci sono nuove tabelle
php artisan migrate --force

```

---

### Perché non fare `update` sul server?

1. **Incoerenza:** Rischieresti di avere la versione 1.2 sul tuo PC e la 1.3 sul server perché uscita nel frattempo.
2. **Risorse:** `composer update` consuma tantissima RAM. Su molti server condivisi o VPS piccole, il processo andrebbe in crash (Out of Memory).
3. **Tempo:** `install` è molto più veloce perché non deve calcolare le dipendenze, deve solo scaricarle.

## Risposta 4

Hai perfettamente ragione: in produzione, il "silenzio" è un rischio. Se un comando fallisce (ad esempio `composer install` finisce la memoria o `php artisan migrate` trova un errore nel SQL), devi saperlo subito senza dover andare a caccia nei log di sistema.

Ecco come evolvere il tuo script Bash per gestire il logging, catturare gli errori e darti un feedback chiaro.

---

### Lo Script di Deploy con Logging e Error Handling

Questo script salva tutto quello che accade in un file dedicato e interrompe l'esecuzione se un comando critico fallisce.

```bash
#!/bin/bash

# Configurazione percorsi e file
PROJECT_PATH="/var/www/tuo-progetto"
LOG_FILE="$PROJECT_PATH/storage/logs/deploy_$(date +%Y-%m-%d).log"
VERSION_FILE="$PROJECT_PATH/storage/app/private/version.json"

# Funzione per loggare messaggi
log_message() {
    echo "$(date '+%Y-%m-%d %H:%M:%S') - $1" >> "$LOG_FILE"
}

# Funzione per gestire i fallimenti
handle_error() {
    log_message "ERRORE CRITICO: $1 fallito. Interruzione deploy."
    # Qui potresti inviare una mail o una notifica Slack/Telegram
    exit 1
}

# Inizio processo
log_message "--- Inizio controllo aggiornamento ---"

# 1. Recupero versioni (assumendo l'uso di jq)
REMOTE_VERSION=$(curl -s https://raw.githubusercontent.com/USER/REPO/main/storage/app/private/version.json | jq -r '.version')
LOCAL_VERSION=$(jq -r '.version' "$VERSION_FILE")

if [ "$REMOTE_VERSION" == "$LOCAL_VERSION" ]; then
    log_message "Nessun aggiornamento necessario (v$LOCAL_VERSION)."
    exit 0
fi

log_message "Nuova versione v$REMOTE_VERSION rilevata. Avvio procedure..."

# 2. Spostamento nella cartella del progetto
cd "$PROJECT_PATH" || handle_error "Accesso alla cartella"

# 3. Pull dal repository
log_message "Eseguo git pull..."
git pull origin main >> "$LOG_FILE" 2>&1 || handle_error "Git Pull"

# 4. Installazione dipendenze
log_message "Eseguo composer install..."
composer install --no-dev --optimize-autoloader --no-interaction >> "$LOG_FILE" 2>&1 || handle_error "Composer Install"

# 5. Migrazioni database
log_message "Eseguo migrazioni database..."
php artisan migrate --force >> "$LOG_FILE" 2>&1 || handle_error "Artisan Migrate"

# 6. Pulizia e ottimizzazione cache
log_message "Ottimizzazione cache Laravel..."
php artisan config:cache >> "$LOG_FILE" 2>&1
php artisan route:cache >> "$LOG_FILE" 2>&1
php artisan view:cache >> "$LOG_FILE" 2>&1

log_message "--- AGGIORNAMENTO COMPLETATO CON SUCCESSO (v$REMOTE_VERSION) ---"

```

---

### Cosa abbiamo aggiunto?

1. **Redirezione dell'Output (`>> "$LOG_FILE" 2>&1`):**

* `>>` aggiunge l'output standard (i messaggi di successo) al file log.
* `2>&1` cattura anche gli **errori** (stderr) e li scrive nello stesso file. Se `composer` esplode, vedrai l'errore esatto nel log.

1. **Operatore `||` (OR):** Se il comando a sinistra fallisce, esegue la funzione `handle_error` a destra.
2. **Blocco di Sicurezza:** Se il `git pull` fallisce, lo script si ferma immediatamente (`exit 1`) per evitare di eseguire migrazioni o installazioni su un codice corrotto o incompleto.

---

### Come leggere il log

Quando vuoi sapere cosa è successo, ti basta leggere il file dal terminale:

```bash
tail -f storage/logs/deploy_2026-03-13.log

```

## Domanda 1

Come faccio il test di federation add, ovvero l'aggiunta di un record

## Risposta 1

Per testare l'aggiunta di un record in Laravel, la via più professionale ed efficace è scrivere un **Feature Test**. Questo ti permette di simulare l'invio del form e verificare che i dati finiscano effettivamente nel database.

Ecco come procedere passo dopo passo:

### 1. Crea il file di test

Esegui questo comando nel terminale per generare la classe di test:

```bash
php artisan make:test FederationTest

```

### 2. Scrivi il test per la creazione

Apri il file appena creato in `tests/Feature/FederationTest.php`. Ecco un esempio completo che copre il tuo scenario (incluse le rotte e i campi in camelCase che abbiamo sistemato):

```php
<?php

namespace Tests\Feature;

use Tests\TestCase;
use App\Models\User;
use App\Models\Federation;
use Illuminate\Foundation\Testing\RefreshDatabase;

class FederationTest extends TestCase
{
    // Questo trait svuota il database (o usa i database in memory) a ogni test
    use RefreshDatabase;

    /** @test */
    public function an_admin_can_store_a_new_federation()
    {
        // 1. Prepariamo un utente autenticato (se la rotta è protetta da middleware 'auth')
        $user = User::factory()->create();
        
        // 2. Dati da inviare (usando i nomi camelCase che hai nel form/rules)
        $formData = [
            'federationName' => 'Federazione Italiana Coding',
            'federationCode' => 'FIC01',
            'description'    => 'Descrizione di prova',
        ];

        // 3. Eseguiamo la chiamata POST alla tua rotta
        $response = $this->actingAs($user)
                         ->post(route('federation-store'), $formData);

        // 4. Verifichiamo che il redirect avvenga correttamente alla index
        $response->assertRedirect(route('federation-index'));

        // 5. Verifichiamo che il record esista nel database
        // Nota: se il DB usa snake_case, qui mappiamo i nomi delle colonne
        $this->assertDatabaseHas('federations', [
            'name' => 'Federazione Italiana Coding',
            'code' => 'FIC01',
        ]);
    }

    /** @test */
    public function federation_creation_requires_a_name()
    {
        $user = User::factory()->create();

        // Inviamo dati vuoti per testare la validazione
        $response = $this->actingAs($user)
                         ->post(route('federation-store'), [
                             'federationName' => '', 
                         ]);

        // Verifichiamo che ci siano errori di validazione per il campo specifico
        $response->assertSessionHasErrors(['federationName']);
    }
}

```

---

### 3. Esegui il test

Puoi eseguire il test direttamente dal terminale:

```bash
php artisan test --filter FederationTest

```

O, se hai configurato il **Task di VS Code** come abbiamo visto prima, puoi mappare questo comando a un'attività rapida.

### 4. Suggerimento per il Database di Test

Assicurati che nel tuo file `phpunit.xml` (nella root del progetto) siano decommentate queste righe per usare un database SQLlite in memoria, così i test saranno velocissimi e non sporcheranno il tuo database di sviluppo:

```xml
<env name="DB_CONNECTION" value="sqlite"/>
<env name="DB_DATABASE" value=":memory:"/>

```

---

### Perché questo test è utile per il tuo "inghippo"?

Se il test fallisce con un errore **422** (Unprocessable Entity) o ti dice che la sessione ha errori, saprai immediatamente che il problema è ancora nelle `rules()` della tua `StoreFederationRequest`. Se invece passa, avrai la certezza matematica che il tuo backend è a posto.

## Risposta 2

Ottima scelta! **Pest** è molto più leggibile e "moderno" rispetto a PHPUnit standard. Se stai usando Laravel 12, Pest è probabilmente già pre-installato o comunque la scelta caldamente raccomandata.

Ecco come scriveremmo lo stesso test per la creazione della Federazione utilizzando la sintassi fluida di Pest.

### 1. Crea il file di test con Pest

Se non lo hai ancora fatto, puoi generarlo con:

```bash
php artisan make:test FederationTest --pest

```

### 2. Scrivi il Test in `tests/Feature/FederationTest.php`

```php
<?php

use App\Models\User;
use App\Models\Federation;
use function Pest\Laravel\{actingAs, post, assertDatabaseHas};

// Utilizza il trait per resettare il database ad ogni test
uses(Tests\TestCase::class, Illuminate\Foundation\Testing\RefreshDatabase::class);

it('allows an admin to create a new federation', function () {
    // Arrange: Prepariamo l'utente e i dati (camelCase come nel tuo form)
    $user = User::factory()->create();
    
    $formData = [
        'federationName' => 'Lega Nazionale Coding',
        'federationCode' => 'LNC26',
        'description'    => 'Descrizione di test',
    ];

    // Act: Eseguiamo la richiesta
    $response = actingAs($user)
        ->post(route('federation-store'), $formData);

    // Assert: Verifichiamo il risultato
    $response->assertRedirect(route('federation-index'));
    
    // Verifichiamo che i dati siano nel DB (mappati nelle colonne snake_case)
    assertDatabaseHas('federations', [
        'name' => 'Lega Nazionale Coding',
        'code' => 'LNC26',
    ]);
});

it('fails validation if federationName is missing', function () {
    $user = User::factory()->create();

    actingAs($user)
        ->post(route('federation-store'), [
            'federationName' => '', // Campo vuoto
        ])
        ->assertSessionHasErrors(['federationName']);
});

```

### 3. Come eseguire i test con Pest

```json
        "panel": "new"
    }

```

*Con questo task, se hai il file del test aperto e premi il comando, VS Code eseguirà solo i test di quel file.*

##  Domanda 4

Quando "è giusto" che esca un errore come faccio a far andar bene il test, ovvero che si verifichi l'errore ma non blocchi la catena dei test?

## Risposta 4

In gergo tecnico, questo si chiama **Testing the Unhappy Path** (testare il percorso "infelice"). Quando un test deve verificare che un errore avvenga, il test è considerato "superato" proprio se l'errore si manifesta.

Pest e Laravel offrono diversi modi per "aspettarsi" un errore senza che questo faccia fallire la suite dei test.

### 1. Testare errori di Validazione (422 / Redirect)

Se il tuo `FederationController` usa le `FormRequest`, Laravel non lancia un'eccezione che rompe il codice, ma effettua un redirect con gli errori in sessione. In questo caso, il test "passa" se verifichi la presenza di quegli errori.

```php
it('non permette di creare una federazione senza nome', function () {
    $user = User::factory()->create();

    $response = actingAs($user)
        ->post(route('federation-store'), [
            'federationName' => '', // Dato mancante
        ]);

    // Il test passa se queste asserzioni sono vere:
    $response->assertStatus(302); // Verifica il redirect automatico
    $response->assertSessionHasErrors(['federationName']); // Verifica l'errore specifico
});

```

### 2. Testare Eccezioni (Exception)

Se invece vuoi testare un errore che lancia un'eccezione vera e propria (ad esempio un `LogicException` o un errore di autorizzazione 403), Pest ha un metodo dedicato: `throws()`.

```php
it('lancia un errore se il codice federazione è duplicato', function () {
    // Supponendo che tu lanci un'eccezione manuale nel controller
    $federation = Federation::factory()->create(['code' => 'TEST1']);

    expect(fn () => (new FederationService)->create(['code' => 'TEST1']))
        ->throws(Exception::class, 'Codice già esistente'); 
        // Il test passa solo se viene lanciata QUESTA eccezione
});

```

### 3. Testare errori di Autorizzazione (403 Forbidden)

Se un utente non loggato o senza permessi prova ad aggiungere un record, il test deve verificare che l'accesso sia negato.

```php
it('impedisce ai non-autenticati di vedere il form', function () {
    // Non usiamo actingAs($user)
    $response = get(route('federation-create'));

    $response->assertRedirect('/login'); 
    // Il test passa perché abbiamo confermato che il sistema è protetto
});

```

---

### Strategia "AssertFail"

Se vuoi testare un fallimento generico in una catena di asserzioni, puoi usare:

* `assertSessionHasErrors()`: Per errori di form.
* `assertStatus(403)` o `404` o `500`: Per verificare che il server risponda con un codice di errore previsto.

**In sintesi:** Il test non si blocca perché tu stai dicendo a Pest: *"Ehi, mi aspetto che questa azione fallisca in questo modo specifico"*. Se fallisce come previsto, il test diventa **verde**.

## Risposta 5

Certamente! Testare l'unicità di un campo (come il codice della federazione) è fondamentale per evitare che il database vada in crash con errori SQL "Integrity constraint violation".

Ecco come scrivere il test con **Pest** per verificare che il sistema blocchi un codice duplicato:

```php
it('impedisce la creazione di una federazione con un codice già esistente', function () {
    // 1. Arrange: Creiamo una federazione già esistente nel database
    $existingFederation = Federation::factory()->create([
        'code' => 'ITA123'
    ]);

    $user = User::factory()->create();

    // 2. Act: Proviamo a crearne un'altra con lo stesso codice
    $response = actingAs($user)
        ->post(route('federation-store'), [
            'federationName' => 'Nuova Federazione',
            'federationCode' => 'ITA123', // <--- Duplicato!
            'description'    => 'Altra descrizione',
        ]);

    // 3. Assert: 
    // Laravel dovrebbe rimandarci indietro (302) con l'errore in sessione
    $response->assertStatus(302);
    $response->assertSessionHasErrors(['federationCode']);
    
    // Verifichiamo che nel database ci sia ancora solo 1 record con quel codice
    expect(Federation::where('code', 'ITA123')->count())->toBe(1);
});

```

### Come interpretare il risultato

* **Se il test è VERDE:** Il tuo codice sta gestendo correttamente l'errore (probabilmente hai una regola `unique:federations,code` nella tua `FormRequest`). Il test non si è bloccato, ha solo confermato che "l'errore previsto è avvenuto".
* **Se il test è ROSSO:** Significa che il record è stato creato due volte (ahi!) o che il server è andato in errore 500 invece di gestire la validazione gentilmente.

### Piccolo trucco: Messaggi di errore personalizzati

Se vuoi essere ancora più pignolo, puoi verificare che il messaggio di errore sia quello esatto che si aspetta l'utente:

```php
$response->assertSessionHasErrors([
    'federationCode' => 'Il codice federazione è già stato utilizzato.'
]);

```

### E se volessi testare un errore fatale (Eccezione)?

Se per qualche motivo vuoi testare che il codice "esploda" (lanciando un'eccezione) anziché fare un redirect, Pest usa la funzione `throws()`:

```php
it('esplode se accade qualcosa di catastrofico', function () {
    expect(function () {
        // Chiami una funzione che sai che deve fallire
        App\Actions\CreateFederation::execute(['dati_corrotti']);
    })->throws(\Exception::class);
});

```
